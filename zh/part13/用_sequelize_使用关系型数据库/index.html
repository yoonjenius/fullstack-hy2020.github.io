    Advantages and disadvantages of document databases

    【文档数据库的优势和劣势】

    我们在之前的章节中一直使用的是MongoDB 数据库。 Mongo 是一个 文档数据库document database ，它其中一个典型的特点是它的 不可知性skepticity 。 比如说， 数据库对存入到自己集合中的数据类型是没有感知的。数据库的schema 仅仅存在于程序的代码中，对数据的解释采用了一种特定的方式，比如，通过一些字段所引用的对象来判断。

    在第三和第四章的例子中，应用中的数据库存放了 notes 和 users。

    notes 的集合存储的notes 类似这样：

    [
  {
    "_id": "600c0e410d10256466898a6c",
    "content": "HTML is easy"
    "date": 2021-01-23T11:53:37.292+00:00,
    "important": false
    "__v": 0
  },
  {
    "_id": "600c0edde86c7264ace9bb78",
    "content": "CSS is hard"
    "date": 2021-01-23T11:56:13.912+00:00,
    "important": true
    "__v": 0
  },
]

    users集合中存储 Users 的类似这样：

    [
  {
    "_id": "600c0e410d10256466883a6a",
    "username": "mluukkai",
    "name": "Matti Luukkainen",
    "passwordHash" : "$2b$10$Df1yYJRiQuu3Sr4tUrk.SerVz1JKtBHlBOARfY0PBn/Uo7qr8Ocou",
    "__v": 9,
    notes: [
      "600c0edde86c7264ace9bb78",
      "600c0e410d10256466898a6c"
    ]
  },
]

    MongoDB 知道字段所存储的类型，但无法知道实体集合，例如user 的ids 是什么类型的。Mongo DB 也不关心字段中集合里的实体的是什么类型。因此MongoDB 将其整个实体交给类程序来确保信息是被正确存放到数据库的。

    这种对schema 无感知的模式有其优势和劣势。其中一个优势是其灵活的schema 不可知性所带来的：由于在数据库层面不需要对数据进行schema 定义，应用开发在某些场景下会加速，而且更简单，只需要很少的努力来定义schema，以及scheme的改变。没有Schema所带来的问题都被延后了：也就是都留给了编程阶段。数据库本身无法判断所存入的数据是否是 合法的。 比如说，是否所有的主键字段都有值，是否引用类型的字段所引用的实体是类型正确的等等。

    The relational databases that are the focus of this section, on the other hand, lean heavily on the existence of a schema, and the advantages and disadvantages of schema databases are almost the opposite compared of the non-schema databases. 关系型数据库所关注的就是这一部分，换句话说，严重依赖schema 信息，因此其优势和劣势刚好与non-schema 的数据库所相反了。

    为什么之前的章节用MongoDB ，就是因为它弱schema 的特性，即使不知道关系型数据库的知识，也能使它用起来更简单。本课程的大多数用例，我自己会选择关系型数据库。

    Model

    使用Sequelize 时，数据库中的每一个表都代表了一个 model ， 也就是一个独立的 JavaScript 类。我们现在定义模型 Note ，代表应用中的 notes 表，将代码改变成如下格式：

    require('dotenv').config()
const { Sequelize, Model, DataTypes } = require('sequelize')const express = require('express')
const app = express()

const sequelize = new Sequelize(process.env.DATABASE_URL, {
  dialectOptions: {
    ssl: {
      require: true,
      rejectUnauthorized: false
    }
  },
});

class Note extends Model {}Note.init({  id: {    type: DataTypes.INTEGER,    primaryKey: true,    autoIncrement: true  },  content: {    type: DataTypes.TEXT,    allowNull: false  },  important: {    type: DataTypes.BOOLEAN  },  date: {    type: DataTypes.DATE  }}, {  sequelize,  underscored: true,  timestamps: false,  modelName: 'note'})
app.get('/api/notes', async (req, res) => {
  const notes = await Note.findAll()  res.json(notes)
})

const PORT = process.env.PORT || 3001
app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`)
})

    代码上有一些注释。Note 的模型定义并没有什么神奇的地方，每个列都有一个类型定义，以及其他的属性，例如是否为该表的主键。模型定义中的第二个参数包括 sequelize 属性，和其他的 配置信息。我们同样定义了该表中不常用的时间戳列（created_at 和 updated_at）。

    我们同样定义了 underscored: true ， 表示表名来自于模型名，并且是 snake case 版本的。实际上也就是说，如果模型的名字叫 notes ，我们这里就是 “Note”，那么代表该 表 的模型就写作小写开头的复数形式，例如 notes 。另一种情况，模型名如果是“两部分” ， 例如 StudyGroup ，那么表名就是study_groups。与直接引用表名不同，Sequelize 允许显示地定义表名。

    同样的命名策略对列也是适用的。如果我们定义了note 的列为 creationYear ， 代表信息创建的年份，我们会将其定义的model 如下所示：

    Note.init({
  // ...
  creationYear: {
    type: DataTypes.INTEGER,
  },
})

    数据库中相关列的名字是creation_year 。在代码中，所引用的列与model中的格式相同，比如说是“驼峰式”的。

    我们同样定义了 modelName: 'note'，默认“model name”是大写的 Note.但是我们想要小写字母开头，需要更方便的 方法。

    使用模型提供的查询接口 query interface ，数据库操作很容易完成， 方法 findAll 的工作按照其名称所假定的那样:

    app.get('/api/notes', async (req, res) => {
  const notes = await Note.findAll()  res.json(notes)
})

    命令行告诉你方法 Note.findAll() 的调用会产生如下查询 ：

    Executing (default): SELECT "id", "content", "important", "date" FROM "notes" AS "note";

    app.use(express.json())

    Creating database tables automatically

    我们的应用现在有一个令人不快的方面，它假设存在具有完全正确的schema的数据库，也就是说，已经使用了适当的 create table 命令创建了 notes 表。

    因为程序代码存储在 GitHub 上，所以把创建数据库的命令也存储在程序代码的上下文中是有意义的，这样数据库schema就肯定和程序代码期望的是一样的。Sequelize 实际上能够通过使用 models 方法 sync 从模型定义自动生成schema。

    我们现在销毁掉数据库，在命令行执行：

    drop table notes;

    利用 \d 命令检查，发现表已经从数据库删除了。

    username=> \d
Did not find any relations.

    应用目前不好使了。

    在model Note 定义后，我们添加如下的命令：

    Note.sync()

    应用启动后，会打印如下信息：

    Executing (default): CREATE TABLE IF NOT EXISTS "notes" ("id" SERIAL , "content" TEXT NOT NULL, "important" BOOLEAN, "date" TIMESTAMP WITH TIME ZONE, PRIMARY KEY ("id"));

    也就是说，当应用启动时，命令 CREATE TABLE IF NOT EXISTS "notes"... 会被执行，如果 notes 不存在会创建一个表。

    Other operations

    让我们为应用完成一些更多的操作。

    查找一个单独的note，可以通过方法 findByPk ， 因为它时通过主键来查询的：

    app.get('/api/notes/:id', async (req, res) => {
  const note = await Note.findByPk(req.params.id)
  if (note) {
    res.json(note)
  } else {
    res.status(404).end()
  }

    查询一个单独的note 会产生如下的SQL命令：

    Executing (default): SELECT "id", "content", "important", "date" FROM "notes" AS "note" WHERE "note". "id" = '1';

    如果找到了note， 会返回 null ，这个用例中相关的状态码就返回了。

    修改note 的动作就完成了。只要修改 important 字段是支持的就行，因为前台并不需要其他的字段：

    app.put('/api/notes/:id', async (req, res) => {

    对于对象集合，toJson方法并不能直接工作，必须为集合中每个对象调用该方法：

    router.get('/', async (req, res) => {
  const notes = await Note.findAll()

  console.log(notes.map(n=>n.toJSON()))
  res.json(notes)
})

    打印结果如下图所示：

    [ { id: 1,
    content: 'MongoDB is webscale',
    important: false,
    date: 2021-10-09T13:52:58.693Z },
  { id: 2,
    content: 'Relational databases rule the world',
    important: true,
    date: 2021-10-09T13:53:10.710Z } ]

    但是，一个更好的解决方案是通过 JSON.stringify: 方法将集合转换成JSON 进行打印：

    router.get('/', async (req, res) => {
  const notes = await Note.findAll()

  console.log(JSON.stringify(notes))
  res.json(notes)
})

    这种方式更好，尤其是当集合中的对象包含其他对象的时候。此外，将屏幕上的对象格式化为更易读的格式是十分有用的，可以通过一下命令实现：

    console.log(JSON.stringify(notes, null, 2))

    打印结果如下所示：

    [
  {
    "id": 1,
    "content": "MongoDB is webscale",
    "important": false,
    "date": "2021-10-09T13:52:58.693Z"
  },
  {
    "id": 2,
    "content": "Relational databases rule the world",
    "important": true,
    "date": "2021-10-09T13:53:10.710Z"
  }
]
