    Migrations

    我们可以像以前一样继续，即更改定义表的模型，并依靠 Sequelize 将更改同步到数据库。这就是文件 models/index.js 中的修改

    const Note = require('./note')

    然而，从长远来看，这种方法是没有意义的。让我们移除执行同步的代码行，转而使用一种更加繁琐的方式—— Sequelize (以及许多其他库)提供的迁移 migrations 。

    实际上，迁移是一个描述对数据库进行某些修改的 JavaScript 文件。一次为每个单个或多个更改创建一个单独的迁移文件。Sequelize 记录哪些迁移已经执行，即迁移引起的哪些更改与数据库模式同步。随着新迁移的创建，Sequelize 将不断更新尚未对数据库模式进行更改的内容。通过这种方式，更改是以受控的方式进行的，程序代码存储在版本控制中。

    首先，创建一个迁移，将数据库带到其当前状态。迁移的代码如下

    const { DataTypes } = require('sequelize')

module.exports = {
  up: async (queryInterface) => {
    await queryInterface.createTable('notes', {
      id: {
        type: DataTypes.INTEGER,
        primaryKey: true,
        autoIncrement: true
      },
      content: {
        type: DataTypes.TEXT,
        allowNull: false
      },
      important: {
        type: DataTypes.BOOLEAN
      },
      date: {
        type: DataTypes.DATE
      },
    })
    await queryInterface.createTable('users', {
      id: {
        type: DataTypes.INTEGER,
        primaryKey: true,
        autoIncrement: true
      },
      username: {
        type: DataTypes.STRING,
        unique: true,
        allowNull: false
      },
      name: {
        type: DataTypes.STRING,
        allowNull: false
      },
    })
    await queryInterface.addColumn('notes', 'user_id', {
      type: DataTypes.INTEGER,
      allowNull: false,
      references: { model: 'users', key: 'id' },
    })
  },
  down: async (queryInterface) => {
    await queryInterface.dropTable('notes')
    await queryInterface.dropTable('users')
  },
}

    迁移文件定义defines 上 up下down函数，其中第一个定义在执行迁移时应该如何修改数据库。down函数再次告诉您如果需要取消迁移，则如何取消迁移。

    我们的迁移包含三个操作，第一个创建一个表 notes，第二个创建一个表 users，第三个为 notes的创建者添加一个引用键。架构中的更改是通过调用 queryInterface 对象方法定义的。

    在定义迁移时，必须记住与模型不同的是，列和表名是用snake模式写的:

    await queryInterface.addColumn('notes', 'user_id', {  type: DataTypes.INTEGER,
  allowNull: false,
  references: { model: 'users', key: 'id' },
})

    因此，在迁移中，表和列的名称完全按照它们进入数据库的方式编写，而模型使用 Sequelize 的默认 camelCase 名称。

    在文件 migrations/2021120900initializenotesand_users.js 中保存迁移代码。Js.迁移文件名应该在字母顺序中，这样以前的更改总是在字母表中新的更改之前。实现这个顺序的一个好方法是用数据和序列号启动迁移文件名。

    我们可以使用 Sequelize 命令行工具 Sequelize command line tool 从命令行运行迁移。但是，我们选择使用 Umzug 库从程序代码手动执行迁移。让我们安装依赖

    使用了两个中间件，第一个名为 tokenExtractor 的中间件与生成note 的路由使用的中间件相同，即它将解码的令牌放置在请求对象字段 decodedToken 中。当第二个中间件 isAdmin 检查用户是否是 admin 时，请求被设置为401，并返回一个适当的错误消息。

    请注意两个中间件是如何链接到路由器的，这两个中间件都是在实际路由处理程序之前执行的。将中间件链接到任意数量的请求连接是可能的。

    中间件 tokenExtractor 现在被移动到 util/middleware.js ，因为它是从多个位置使用的。

    const jwt = require('jsonwebtoken')
const { SECRET } = require('./config.js')

const tokenExtractor = (req, res, next) => {
  const authorization = req.get('authorization')
  if (authorization && authorization.toLowerCase().startsWith('bearer ')) {
    try {
      req.decodedToken = jwt.verify(authorization.substring(7), SECRET)
    } catch{
      return res.status(401).json({ error: 'token invalid' })
    }
  } } else {
    return res.status(401).json({ error: 'token missing' })
  }
  next()
}

module.exports = { tokenExtractor }

    Many-to-many connections

    继续扩展应用程序，以便将每个用户添加到一个或多个团队 teams中。

    由于任意数量的用户可以加入一个团队，而一个用户可以加入任意数量的团队，因此我们处理的是多对多 many-to-many、多对多类型的连接，这种连接通常是在关系数据库中使用连接表connection table实现的。

    Eager vs lazy fetch

    当我们使用 include 属性进行查询时:

    User.findOne({
  include: {
    model: note
  }
})

    所谓的即时获取eager fetch，即通过连接查询将表的所有行连接到用户，例如，用户记录的笔记，同时从数据库中获取。这通常是我们所希望的，但是也有一些情况需要执行所谓的延迟获取lazy fetch，即只在需要的时候搜索与用户相关的团队。

    现在，让我们修改一个用户路由的路由，这样只有在查询参数团队teams为请求设置的情况下，它才能获取用户的团队:

        model: note,
        attributes: { exclude: ['userId'] }
      },
      {
        model: Note,

        attributes: { exclude: ['userId']},
        through: {
          attributes: []
        },
        include: {
          model: user,
          attributes: ['name']
        }
      },
    ]
  })

  if (!user) {
    return res.status(404).end()
  }

  // hightlight-start
  let teams = undefined

  if (req.query.teams) {
    teams = await user.getTeams({

    难道我们不能优化代码，例如，模型可以导出共享的部分用于迁移吗？

    然而，问题是模型的定义可能会随着时间的推移而改变，例如字段名称可能会改变名称或者它的数据类型可能会改变。迁移必须能够在从开始到结束的任何时候成功地执行，如果迁移依赖于模型具有某些内容，那么在一个月或一年的时间内它将不再是真实的。因此，尽管有“复制粘贴”，迁移代码应该与模型代码完全分离。

    一个解决方案是使用 Sequelize 的命令行工具 command line tool，该工具根据命令行给出的命令生成模型和迁移文件。例如，下面的命令将创建一个名称、用户名和管理属性的 User 模型，以及管理创建数据库表的迁移:

    npx sequelize-cli model:generate --name User --attributes name:string,username:string,admin:boolean

    在命令行中，您还可以运行以及回滚，即取消迁移。不幸的是，命令行文档很少，在本课程中，我们决定手动执行模型和迁移。这个解决方案可能是明智的，也可能不是。

    Grande finale: 在第4部分的末尾 towards the end of part 4 提到了一个令牌临界问题: 如果一个用户对系统的访问被决定撤销，那么用户仍然可以使用占有的令牌来使用系统。

    通常的解决方案是在后端数据库中存储发给客户的每个令牌的记录，并检查每个请求中的访问是否仍然有效。在这种情况下，如果需要，可以立即删除令牌的有效性。这种解决方案通常称为服务器端会话。

    现在扩展系统，以便失去访问权限的用户将不能执行任何需要日志记录的操作。

    为了实现，您可能至少需要以下内容

    • 用户表中的布尔值列，以指示是否禁用 ID

      • 从数据库直接禁用和启用 id 就足够了

    • 一个能记住活跃会话的表

      • 当用户进行登录(即操作 POST/api/login)时，session 存储在表中
      • 当用户进行登录操作时，总是检查会话的存在(和有效性)
    • 一个允许用户“注销”系统的路由，即实际上从数据库中删除活动会话，该路由可以是 DELETE/api/logout

    请记住，登录不应成功与“过期令牌”，即与相同的令牌登出后。

    您还可以选择使用一些特定构建的 npm 库来处理函数。
